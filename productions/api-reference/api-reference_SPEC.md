# API Reference 文档规范

本规范适用于 `productions/api-reference/preview/{zh,en}` 下的 API Reference 文档，用于指导大模型自动生成或更新 Preview 文档。根目录 `SPEC.md` 是上位规范；若本文件与根规范冲突，以根规范为准。本文件只补充 API Reference 子文档库的组织、书写和排版要求。

## 内容范围

- 只分析和更新 `preview`；`archives` 不参与当前 API 参考生成。
- API Reference 描述 ADK API Server 的公开 HTTP 能力，包括应用、会话、运行、制品、记忆、评测和调试路由。
- 文档面向需要直接调用 HTTP API、对接 SDK 或排查请求结构的开发者。
- 不记录内部路由注册方式、服务实现类、源码路径或内部管理接口。
- 中英文内容必须路径一致、层级一致、接口覆盖一致。


`docs.json` 中 API Reference 当前位于 `ADK API Server` 分组。新增页面或调整顺序时，必须同步更新中英文导航入口。

## 页面职责

- `index.mdx`：说明基础路径、通用路径参数、服务信息路由和全局约定。
- `sessions.mdx`：应用列表、应用详情、会话创建、读取、列表、状态更新与删除。
- `run.mdx`：一次性运行与流式运行，覆盖请求体、消息结构和调用示例。
- `artifacts.mdx`：制品保存、读取、删除、版本和元数据。
- `memory.mdx`：写入会话到记忆等记忆相关接口。
- `evaluation.mdx`：评测相关路由、请求字段和结果读取方式。
- `debug.mdx`：调试和可视化相关路由。

同一主题下接口较少时，可用路由表集中列出；接口字段复杂时，应分出请求体、响应体和查询参数表。

## API 条目写法

API 页面优先使用清晰的参考结构，不写教程式长篇叙述。每个接口或接口组应包含以下信息：

1. 方法与路径，例如 `GET /apps/{app_name}`。
2. 一句话说明用途。
3. 路径参数、查询参数、请求体字段、响应结构。
4. 必要的状态码、错误条件或限制。
5. 对复杂接口给出最小 `curl` 示例。

| --- | --- | --- |

如能确认是否必填，可在说明中直接写“必填”“不提供时……”。不要新增未核实的“是否必填”列。

## 路径与字段格式

- HTTP 方法使用大写 `GET`、`POST`、`PATCH`、`DELETE`。
- 路径参数使用花括号，例如 `/apps/{app_name}/users/{user_id}`。
- 字段名、路径、类型、枚举值、请求体名称和响应体名称使用反引号。
- 类型使用简洁的 OpenAPI 风格：`string`、`boolean`、`integer`、`number`、`object`、`array`。
- 对外可见的数据结构名可以保留，例如 `types.Content`、`types.Part`、`ArtifactVersion`。
- 时间字段必须说明单位，例如 Unix 时间戳（秒）。

## 页面结构

- YAML frontmatter 只包含 `title`。
- 页面首段直接说明该页覆盖的资源或命名空间。
- 二级标题用于资源组，例如 `## 应用`、`## 会话`、`## 一次性运行`。
- 三级标题用于具体接口，例如 `### 列出应用`、`### 创建会话`。
- API Reference 项目尽量少用 Mintlify 组件；只有全局约定或重要限制使用 `<Note>`，危险或破坏性行为使用 `<Warning>`。
- 不添加“下一步”章节。

## 示例

- 示例只在能帮助读者理解请求结构时添加，不必给每个简单 GET 路由都配示例。
- 示例命令使用 `bash` 代码块，优先用 `curl`。
- 示例中基础地址用占位符或本地开发地址，不写内部域名。
- 鉴权头、用户 ID、应用名和会话 ID 使用占位符。
- 流式接口示例需要说明返回事件流或流式响应的行为，不要只复制普通 JSON 示例。

## 事实来源与校验

更新 API Reference 前必须核对公开接口来源：

- 当前 ADK API Server 的公开路由、OpenAPI 或等价机器可读定义。
- 公开请求/响应模型和字段默认行为。
- 可运行的本地服务或测试示例。

不得根据旧文档或字段名猜测请求体、响应体、默认值或错误条件。无法确认的字段不要写入；如果接口存在但字段未完全确认，应先补充事实来源再更新文档。

## 更新检查

提交前至少检查：

- 所有路径、方法、路径参数和查询参数与公开接口一致。
- 请求体和响应体字段覆盖完整，类型准确。
- 中英文页面等价。
- `docs.json` 导航已同步。
- 内部实现细节、源码路径和内部服务名未出现在正文。
- 示例不包含真实密钥、内部域名或用户数据。
